# 自社コントロール改善案

株式帝国では、現状の自社は「会社名・ティッカー・初期株価を決めて上場する」だけで、その後にプレイヤーが経営判断をする余地がほとんどありません。ここを改善すると、単なる株売買ゲームから「自分の会社を育てながら市場で戦うゲーム」に変えられます。

この案は、現在の構成である GitHub Pages + Firebase Realtime Database のまま、無料で実装しやすい順にまとめています。

## まず入れたい最小機能

最初に入れるべきなのは、自社に対して毎日または一定時間ごとに実行できる「経営アクション」です。

プレイヤーは資金を使って、自社の成長率・安定性・イベント耐性を少しずつ変えられるようにします。複雑にしすぎず、まずは以下の3つで十分です。

| アクション | 費用 | 効果 | 狙い |
|---|---:|---|---|
| 研究開発 | 100,000円 | 上昇イベントや自然成長が少し強くなる | 攻めの投資 |
| 広告宣伝 | 80,000円 | 短時間だけ株価に上向き補正 | 即効性 |
| リスク管理 | 70,000円 | 悪材料イベントの下落を軽減 | 守りの投資 |

これだけでも、プレイヤーは「手元資金を株購入に使うか、自社強化に使うか」を選べるようになります。

## 自社ステータス

自社に以下のステータスを追加します。

| ステータス | 意味 | 株価への影響 |
|---|---|---|
| 成長力 | 会社の伸びやすさ | 通常時の株価ドリフトが少し上向く |
| 知名度 | 市場からの注目度 | 広告や好材料イベントの効果が上がる |
| 安定性 | 不祥事・下落への強さ | 悪材料イベントの下落幅が小さくなる |
| 資金力 | 会社内部の余力 | 大きな施策や回復アクションに使う |

Firebase の会社データには、例えば次のように保存します。

```json
{
  "companyStats": {
    "growth": 1,
    "brand": 1,
    "stability": 1,
    "cash": 0,
    "lastActionAt": 1779160000000
  }
}
```

## 画面案

「会社設立」タブを、会社を持っている場合は「自社経営」画面に変えます。

表示する内容は以下です。

| 表示項目 | 内容 |
|---|---|
| 自社情報 | 会社名、ティッカー、現在株価、時価総額風の数値 |
| 経営ステータス | 成長力、知名度、安定性、資金力 |
| 経営アクション | 研究開発、広告宣伝、リスク管理 |
| 最近の施策履歴 | いつ何を実行したか |
| 次回実行可能時間 | 連打防止のクールダウン |

最初はグラフや複雑な UI は不要です。ボタン3つとステータス表示だけでも、ゲームとしてかなり触れるようになります。

## 株価への反映方法

現在の共通株価は Firebase の `market/state` を使っており、誰か1人のブラウザが市場リーダーとして株価を更新しています。この仕組みに、自社ステータスを読み込ませれば無料のまま実装できます。

例として、自社株の更新時だけ以下の補正を入れます。

```js
const growthBonus = companyStats.growth * 0.0005;
const brandBonus = companyStats.brand * 0.0003;
const stabilityBonus = companyStats.stability * 0.0004;
```

好材料イベントでは `growth` と `brand` が高いほど上昇しやすくし、悪材料イベントでは `stability` が高いほど下落しにくくします。

重要なのは、強化しすぎると自社株だけが壊れるので、1回のアクション効果は小さくすることです。プレイヤーが積み重ねた結果として差が出る程度がよいです。

## クールダウン制

無料構成では、処理をなるべく軽くして、不正や連打も抑えたいです。そのため、経営アクションにはクールダウンを入れます。

おすすめは以下です。

| ルール | 内容 |
|---|---|
| 実行間隔 | 1アクションにつき10分に1回 |
| 1日の上限 | 合計10回まで |
| 費用 | 回数が増えるほど少し高くなる |

最初は「10分に1回」だけでも十分です。

## 実装ステップ

実装は段階的に進めるのがおすすめです。

1. 自社データに `companyStats` を追加する
2. 自社経営画面にステータス表示と3つのボタンを追加する
3. ボタン実行時に残高を減らして `companyStats` を更新する
4. 共通株価更新ロジックで、自社の `companyStats` を株価変動に反映する
5. クールダウンと実行履歴を追加する
6. バランス調整をする

## おすすめの最初の実装

最初の実装では、以下の仕様がちょうどよいです。

| 項目 | 内容 |
|---|---|
| 研究開発 | 100,000円。成長力 +1 |
| 広告宣伝 | 80,000円。知名度 +1。次の5分だけ自社株に上向き補正 |
| リスク管理 | 70,000円。安定性 +1 |
| クールダウン | 全アクション共通で10分 |
| ステータス上限 | 各20 |

この仕様なら、実装が軽く、ユーザーにも分かりやすく、ゲーム性もすぐ増えます。

## 将来的な発展案

余裕があれば、次のような拡張もできます。

| 機能 | 内容 |
|---|---|
| 決算発表 | 1日1回、自社の業績イベントが起きる |
| 配当 | 自社株を持つユーザーに少額還元 |
| 新株発行 | 自社資金を増やす代わりに株価が下がる |
| 自社株買い | 資金を使って株価を支える |
| 買収 | 他プレイヤー企業へ影響を与える高額アクション |
| ランキング | 時価総額、成長力、総資産で競う |

## 優先順位

最優先は「経営アクション3つ + ステータス4つ」です。

これを入れるだけで、自社を持つ意味がはっきりします。その後に、決算・配当・ランキングを足すと、長く遊べる方向に伸ばせます。

## 注意点

現在の Firebase ルールは認証済みユーザーなら広く書き込み可能です。経営アクションを本格的に入れる場合、本来は Firebase セキュリティルールで「自分の会社だけ更新できる」「残高を勝手に増やせない」ようにした方が安全です。

ただし、まず遊べる形を無料で作るなら、クライアント側実装から始めて、あとでルール強化する進め方で問題ありません。
